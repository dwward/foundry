                                                    Dash applications

                                                    Published Dash applications are subject to a 30-second timeout, meaning that the Dash server must start up within 30 seconds of executing your Dash application Python file. Otherwise, your Dash application will fail to start up.

                                                    To create a Dash â application, follow the instructions below:

                                                    1. Open the Applications tab in a JupyterLabÂ® workspace.
                                                    2. Select Publish application , choose a location in your Files and Projects for your new application, and provide a module name and variable name. A file will be automatically created at the path corresponding to the provided module name.
                                                    3. Optional: Configure advanced settings. By default, the application settings will match the settings of the JupyterÂ® workspace.
                                                    4. Select Publish and sync to register your new Dash application and sync the code to the backing code repository. After CI checks and publishing are complete, you can select the link in the Applications panel to view the published application.

                                                    The module name of the Dash application should be entered as a dotted path, relative to the root of the repository (for example, app , or if your module is inside a folder, folder.app ). The variable name is the Python variable within the module file that is declared for the Dash app server, which is usually called server .

                                                    To develop locally against an existing Dash application:

                                                    1. Open a terminal window and execute your newly created file, for example, python app.py .
                                                    2. Select Preview in the Dash section of the Applications tab to preview the application.

                                                    You can ensure the Conda/PyPI environment in your JupyterLabÂ® workspace will be automatically restored in your published Dash application using managed Conda/PyPI environments in JupyterÂ® Notebooks .

                                                    Streamlit applications

                                                    Published Streamlit applications are subject to a 30-second timeout, meaning that the Streamlit server must start up within 30 seconds of executing your Streamlit application Python file. Otherwise, your published Streamlit application will fail to start up.

                                                    To create a Streamlit â application, follow the instructions below:

                                                    1. Select Publish application , choose a location in your Files and Projects for your new application, and enter the name of your Python file relative to the root of the repository, for example, app.py .
                                                    2. Optional: Configure advanced settings. By default, the application settings will match the settings of the JupyterLabÂ® workspace.
                                                    3. Select Publish and sync to register your new Streamlit application and sync the code to the backing code repository. After CI checks and publishing are complete, you can select the link in the Applications panel to view the published application.

                                                    To develop locally against an existing Streamlit application:

                                                    1. Open a terminal window and run the code with Streamlit using streamlit run app-name.py .
                                                    2. Open the Applications tab in a JupyterLabÂ® code workspace and select Preview in the Streamlit section to preview the application you are developing locally.

                                                    You can ensure the Conda/PyPI environment in your JupyterLabÂ® workspace will be automatically restored in your published Streamlit application using managed Conda/PyPI environments in JupyterÂ® Notebooks .

                                                    JupyterÂ® Notebook transforms

                                                    Jupyter Notebook transforms currently do not support making API calls to external services, even if the network policy was added in the code workspace. To run reproducible pipelines making API calls to the Internet, you should use

                                                    1. Follow the interface to configure your transform. Code Workspaces will infer the JupyterÂ® notebook used along with the inputs added to produce the output dataset. If the inference is wrong, you can manually edit the inputs and outputs.
                                                    1. Select Sync and run to save the transform configuration and trigger CI checks for publishing the transform and JobSpecs on the output dataset(s).

                                                    Once CI checks are completed, your transform is ready for build. You can use other Foundry data pipeline to which it belongs.

                                                    The transform does not automatically use the environment of the Code Workspace. You should install all the packages needed at runtime at the top of the script file. The code snippet to install a package can be found in the Packages panel. For example, to use scikit-learn and scikit-image in your transform, add this code snippet to the top of the file: 

                                                    !mamba install -y -q scikit-learn scikit-image

                                                    Managed Conda/PyPI environments in JupyterÂ® Notebooks

                                                    Managed Conda/PyPI environments are generally available.

                                                    JupyterÂ® Conda/PyPI environments can be consistently restored across JupyterÂ® workspaces, applications, transforms, and model outputs, using the managed environment feature.

                                                    Managed environments are consistent across Code Workspace workflows, eliminating the need to re-run the same mamba and pip commands every time you restart your workspace, start a application, or run a JupyterÂ® Notebook transform.

                                                    With a managed environment, installed libraries in your working environment are tracked through files in your /home/user/repo folder, allowing the environment to be restored to its original state across workflows.

                                                    The following two files are used to track your managed environment:

                                                    1. /home/user/repo/.envs/maestro/meta.yaml : Known as the environment manifest , it contains the set of requested libraries and versions for your environment. Learn more about the meta.yaml file .
                                                    2. /home/user/repo/.envs/maestro/hawk.lock - Known as the environment lock file , it contains the set of resolved libraries and versions for your environment, solved from the constraints specified in the manifest. The hawk.lock file replaces conda.lock , which has been deprecated. Learn more about the hawk.lock file .

                                                    We do not recommend modifying the hawk.lock lock file. Instead, rely on installation commands to update the files on your behalf. Unless there is a git conflict, you do not need to modify the meta.yaml file either; it will automatically update when packages are interactively installed.

                                                    Your managed environment is installed in your /home/user/envs/default folder. This environment is restored before the following stages:

                                                    • Starting up your JupyterÂ® workspace
                                                    • Starting up applications produced from your workspace
                                                    • Running JupyterÂ® Notebook transforms produced from your workspace
                                                    • Running models

                                                    Usage

                                                    To enable this feature, toggle the Managed Conda environments experimental feature in your workspace settings, located on the left sidebar. Then, restart the workspace:

                                                    Once the feature is enabled, a managed environment named default will be available for you to use:

                                                    To update your managed environment, navigate to the Packages tab in the left sidebar and follow the instructions to add new packages.

                                                    When installing a package through the Packages tab, the install command is executed directly in the JupyterÂ® terminal, rather than executing it within your notebook or application code. Every time an install command is run, the meta.yaml manifest will be updated with your requested package, and the hawk.lock file will reflect the packages installed in the current environment.

                                                    View currently installed packages in your managed environment

                                                    Installed packages are visible in the Packages tab:

                                                    View environment files in the JupyterLabÂ® interface

                                                    Navigate to the View dropdown menu in the workspace and select Show Hidden Files .

                                                    Then, navigate to the .envs/maestro folder to view the meta.yaml and hawk.lock files.

                                                    Synchronize your managed environment with changes to the hawk.lock file

                                                    After checking out a new branch or pulling new git changes, the contents of your hawk.lock file may change.

                                                    To synchronize your managed environment, execute the following command in your workspace terminal: 

                                                    maestro env install

                                                    maestro is the command line interface that updates managed environments. The above command re-resolves the lock file from the manifest and installs the environment into the /home/user/envs/default folder.

                                                    Resolve git conflicts to your environment files

                                                    To resolve git conflicts in your managed environment, first delete the hawk.lock file. Navigate to the meta.yaml file to resolve any conflicts, then merge your branch.

                                                    After merging, run the maestro env install command in your terminal to regenerate the hawk.lock file and reinstall your managed environment.

                                                    Upgrade your managed environment

                                                    To update your managed environment, first remove the hawk.lock file. Then, delete your /home/user/envs/default folder ( rm -rf /home/user/envs/default ). Finally, run the maestro env install command in your terminal to regenerate your hawk.lock file. Your environment will be re-installed with upgraded package versions.
